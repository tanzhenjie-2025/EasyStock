
# 仓库开发/维护提醒 (Remind)
## 一、基础规范
### 1. 代码提交
- 提交信息格式：`[类型][模块]: 具体描述`
  - 类型可选：feat(新功能) / fix(修复bug) / docs(文档) / style(格式) / refactor(重构) / test(测试) / chore(杂项)
  - 示例：`[feat][user-api]: 新增用户登录接口`
- 提交前务必：
  - 本地自测通过，无语法错误/逻辑漏洞
  - 执行代码格式化（如prettier、eslint等）
  - 提交范围最小化，一个提交对应一个独立变更

### 2. 分支管理
- 主分支：`main/master`（仅用于发布正式版本，禁止直接提交）
- 开发分支：`dev`（日常开发合并分支，需通过PR/MR审核）
- 功能分支：`feature/xxx`（从dev分支创建，完成后合并回dev）
- 修复分支：`bugfix/xxx`（从dev分支创建，修复后合并回dev；生产紧急修复从main创建，修复后同步到dev）
- 分支命名：小写+短横线，禁止特殊字符

### 3. 环境配置
- 开发环境：`.env.dev`（本地开发使用，请勿提交到仓库）
- 测试环境：`.env.test`（测试服务器配置，需脱敏）
- 生产环境：`.env.prod`（仅运维配置，仓库中仅保留模板，填充占位符）
- 敏感信息（密钥、token、数据库密码等）：
  - 禁止硬编码到代码
  - 通过环境变量/配置中心管理
  - 仓库中仅保留示例（如`API_KEY=your_api_key_here`）

## 二、开发注意事项
### 1. 依赖管理
- 安装依赖：优先使用仓库指定的包管理器（如`npm`/`yarn`/`pnpm`/`pip`），禁止混用
- 版本锁定：提交`package-lock.json`/`yarn.lock`/`requirements.txt`等锁文件，确保环境一致性
- 依赖更新：定期检查第三方依赖安全漏洞（如`npm audit`/`pip audit`），及时更新

### 2. 代码质量
- 遵循仓库编码规范（如ESLint/PEP8/Alibaba Java Coding Guidelines等）
- 核心逻辑必须添加注释，复杂函数需写函数说明
- 新增功能需配套单元测试，测试覆盖率不低于[XX]%

### 3. 文档维护
- 新增核心功能需同步更新README.md/使用文档
- 接口变更需更新接口文档（如Swagger/OpenAPI）
- 部署流程变更需更新部署文档

## 三、紧急情况处理
### 1. 生产环境bug
- 优先创建紧急修复分支`hotfix/xxx`，从main分支拉取
- 修复后先合入main分支发布紧急版本，再同步到dev分支
- 修复完成后需补充测试用例，避免复现

### 2. 仓库权限/安全问题
- 发现权限泄露：立即联系仓库管理员撤销权限，排查敏感信息泄露风险
- 发现恶意提交/漏洞：立即暂停仓库访问，回滚异常提交，修复漏洞后恢复

### 3. 构建/部署失败
- 先检查构建日志，定位失败原因（依赖缺失/代码语法错误/配置错误等）
- 本地复现问题后修复，确认无误后重新触发构建/部署
- 若为环境问题，同步更新部署文档并通知运维/团队成员

## 四、其他提醒
- 每日下班前确认：本地分支已push，未完成代码已提交（草稿PR/MR）
- 每周同步：清理无用分支（已合并的feature/bugfix分支）
- 定期备份：核心配置/数据需定期备份，避免丢失
- 团队协作：重大变更前需同步团队成员，避免冲突

## 五、bill 订单模块（Django后端）专属开发规范
### 1. 代码结构规范
- 固定目录：`bill/views.py` 所有视图函数、`bill/urls.py` 路由、`bill/models.py` 数据模型
- 路由必须配置命名空间 `app_name = 'bill'`，禁止硬编码URL，使用 `reverse` 反向解析
- 权限常量统一在 `views.py` 顶部定义，禁止分散硬编码

### 2. 模型（Models）开发规范
1. **索引强制要求**
   - 严格使用已定义的联合索引/覆盖索引，禁止随意删除/修改模型索引
   - 高频查询字段（商品拼音、订单状态、创建时间、客户区域）必须保留索引
2. **业务逻辑禁止**
   - 禁止在 `OrderItem.save()` 中编写**库存扣减/恢复**逻辑，统一在视图层批量处理
   - 订单号自动生成逻辑禁止修改，高并发下必须保留 `select_for_update` 锁机制
3. **字段规范**
   - 订单状态固定：`pending(未打印)/printed(已打印)/cancelled(作废)/reopened(重开)`，禁止新增无文档状态
   - 金额字段统一使用 `DecimalField`，禁止用浮点数避免精度丢失

### 3. 视图（Views）开发规范
1. **数据库性能优化**
   - 关联查询必须使用 `select_related`（外键）/`prefetch_related`（多对多），彻底禁止N+1查询
   - 批量操作优先使用 `bulk_create`/`bulk_update`，禁止循环单条写入数据库
   - 分页固定使用 `Paginator`，每页默认10条，禁止全表查询返回数据
2. **事务强制要求**
   - 订单创建、作废、批量结清、库存变更**必须包裹 `transaction.atomic()` 事务**
   - 事务内操作原子化，失败自动回滚，保证数据一致性
3. **缓存使用规范**
   - 固定使用顶部定义的缓存常量：`CACHE_STOCK_LIST`/`CACHE_ORDER_LIST` 等，禁止硬编码缓存时间
   - 缓存页面使用 `@cache_page`，接口缓存使用 `cache.set/get`，缓存键必须唯一且语义化
   - 订单数据变更（创建/作废/结清）后，**手动清理对应缓存**，避免数据不一致

### 4. 权限与安全规范
1. **RBAC权限控制**
   - 所有订单操作必须添加权限装饰器：`@login_required`/`@permission_required`/`@ajax_permission_required`
   - 权限校验严格遵循角色规则：超级管理员 > 管理员 > 操作员
   - 操作员作废订单：仅允许自己创建、5分钟内的订单，禁止越权操作
2. **接口安全**
   - AJAX接口统一使用自定义装饰器，未登录/无权限返回标准JSON格式
   - 敏感操作（作废、结清、批量操作）仅允许POST请求，禁止GET请求

### 5. 业务逻辑核心规范
1. **库存管理**
   - 开单：批量扣减库存，校验库存充足
   - 作废订单：批量恢复库存，禁止单条循环更新
   - 库存数据禁止手动修改，仅通过订单流程变更
2. **订单操作规则**
   - 已打印/已结清订单：禁止作废
   - 仅作废订单允许重开，重开订单状态固定为 `reopened`
   - 结清/撤销结清必须填写备注，日志留痕
3. **日志规范**
   - 所有核心操作（创建/作废/结清/重开）必须调用 `create_operation_log` 记录日志
   - 日志包含：操作人、操作对象、操作详情、IP地址

### 6. 接口与工具规范
1. **搜索功能**
   - 商品/客户搜索统一使用**前缀匹配**（`istartswith`），命中索引，禁止全表模糊查询
   - 搜索接口必须加缓存，减少数据库压力
2. **销售汇总**
   - 每日汇总使用 `generate_daily_summary`/`auto_summary_yesterday` 工具函数
   - 汇总数据禁止手动篡改，仅通过系统自动生成/手动补录

### 7. 测试与部署规范
- 测试重点：订单创建、库存扣减、作废恢复、权限校验、缓存失效
- 部署前：清空测试缓存，校验数据库索引完整性
- 生产环境：禁止直接修改订单数据，所有操作通过系统接口完成

## 六、accounts 用户模块（RBAC权限系统）专属开发规范
### 1. 代码结构规范
- 固定目录：`accounts/views.py`（用户/角色/权限视图）、`accounts/models.py`（RBAC核心模型）、`accounts/urls.py`（权限系统路由）
- 权限/角色常量**统一在models.py顶部定义**，禁止分散硬编码
- 工具函数（`get_client_ip`/`create_operation_log`）仅在accounts模块维护，全局复用

### 2. 模型（Models）开发规范
1. **固定角色强制规则**
   - 系统仅支持3个固定角色：`super_admin`(超级管理员)、`admin`(管理员)、`operator`(开单人)
   - **禁止删除/修改角色编码**，修改角色编码会直接导致权限系统崩溃
   - 角色删除/保存方法已做保护，禁止绕过模型校验操作
2. **权限模型规范**
   - 权限分类固定：订单/商品/用户/客户/系统，禁止新增无文档分类
   - 权限编码唯一，必须使用`小写+下划线`格式，禁止随意修改
   - 高频校验索引`code+is_active`禁止删除，保证权限校验性能
3. **用户模型规范**
   - 继承`AbstractUser`，**彻底移除Django原生权限/分组**，仅使用自定义RBAC
   - `user_code`(用户编号)唯一且必填，作为开单人追责核心字段
   - 金额/统计相关字段统一使用`Decimal`，禁止浮点数
   - 用户表索引`user_code/phone/is_active`禁止修改，优化用户查询

### 3. 视图（Views）开发规范
1. **权限装饰器强制使用**
   - 所有后台页面必须添加`@login_required`+`@permission_required`双装饰器
   - 权限编码严格与models.py常量一致，禁止手写硬编码
2. **数据库性能优化**
   - 用户查询必须使用`select_related('role')`，消除N+1查询
   - 用户列表/详情统计使用`annotate`聚合查询，禁止循环统计
   - 分页固定使用`Paginator`，每页10条，禁止全表返回
3. **敏感操作规范**
   - 用户启用/禁用、密码重置、编辑仅支持**POST请求**
   - 超级管理员专属操作（角色权限配置）禁止普通角色访问

### 4. 缓存使用规范
1. **固定缓存规则**
   - 角色缓存：`system_roles_all`（1天），权限缓存：`role_perms_*`（30分钟）
   - 用户权限缓存：`user_perm_{用户ID}_{权限编码}`（1小时）
2. **缓存清理强制要求**
   - 修改用户角色/密码/状态 → 清理对应用户权限缓存
   - 修改角色权限 → 清理角色缓存+全量用户权限缓存
   - 新增/编辑用户 → 清理全局角色缓存
3. **缓存禁止**
   - 登录/登出/密码修改等敏感接口**禁止缓存**

### 5. RBAC权限核心规范
1. **角色权限边界**
   - 超级管理员：拥有所有权限，无需配置，自动放行
   - 管理员：拥有订单/商品/客户/日志管理权限，可作废他人订单
   - 开单人：仅基础开单、查看、作废自己订单（5分钟内）权限
2. **权限校验方法**
   - 单权限校验：`user.has_permission(权限编码)`
   - 多权限校验：`user.has_any_permission(编码1,编码2)`
   - 禁止手写权限判断逻辑，统一调用模型方法
3. **权限初始化**
   - 系统权限/角色通过`post_migrate`信号自动初始化，禁止手动SQL插入
   - 部署/迁移后自动同步权限，无需手动配置

### 6. 安全与日志规范
1. **登录安全**
   - 密码重置后强制用户修改密码（`force_password_change=True`）
   - 登录/登出/改密均记录IP+操作日志
   - 禁用用户禁止登录系统
2. **日志规范**
   - 操作类型常量统一使用：`OP_TYPE_LOGIN/LOGOUT/RESET_PASSWORD`等
   - 所有用户操作（增删改查、密码重置、状态切换）必须调用`create_operation_log`
   - 日志自动记录操作人、IP、时间、详情，不可删除/篡改
3. **IP获取**
   - 统一使用`get_client_ip`获取客户端IP，禁止手写获取逻辑

### 7. 业务逻辑核心规则
1. **用户管理**
   - `user_code`全局唯一，禁止重复
   - 超级管理员账户禁止禁用/删除
   - 密码长度至少8位，强制复杂度校验
2. **角色权限配置**
   - 仅超级管理员可进入角色权限配置页面
   - 禁止给角色配置空权限（超级管理员除外）
3. **个人资料**
   - 用户可修改个人信息/密码，禁止修改角色/用户编号

### 8. 测试与部署规范
- 测试重点：登录鉴权、权限拦截、密码重置、缓存清理、角色边界
- 部署前：执行迁移命令，自动初始化角色+权限
- 生产环境：禁止直接修改`auth_user`/角色/权限表数据
- 权限变更后：清理缓存，重新登录生效

## 七、area 区域&区域组管理模块专属开发规范
### 1. 代码结构规范
- 固定目录：`area/views.py`（区域/区域组CRUD）、模板目录`area_manage/`、关联模型`Area/AreaGroup`
- **常量统一置顶**：分页`AREA_PAGE_SIZE`、排序白名单`ALLOW_SORT_AREA/GROUP`、缓存配置、缓存前缀
- 工具函数复用：`format_datetime`统一时间格式化，禁止在视图中重复写时间格式化逻辑
- 缓存工具函数：`generate_cache_key`/`clear_area_cache`/`clear_group_cache` 全局复用，禁止手写缓存清理逻辑

### 2. 模型（Models）开发规范
1. **关联关系规范**
   - `Area`（区域）与`AreaGroup`（区域组）为**多对多关联**，禁止修改关联关系
   - 区域关联客户`Customer`、订单`Order`，删除区域前自动校验关联数据
2. **查询优化强制要求**
   - 所有单表查询必须使用`.only()`指定需要字段，禁止全字段查询
   - 多对多查询使用`prefetch_related`优化，彻底禁止N+1查询
   - 统计数据使用数据库`annotate/Subquery`聚合，禁止Python循环统计
3. **字段规范**
   - 区域/组名称`name`唯一，禁止重复
   - 时间字段统一使用`create_time/update_time`，禁止混用字段

### 3. 视图（Views）开发规范
1. **数据库性能优化**
   - 列表查询：仅查询必要字段 + 排序白名单校验 + 分页限制
   - 详情查询：`only()`+`prefetch_related`预加载关联数据
   - 统计查询：使用`get_area_statistics`/`get_group_statistics`工具函数
2. **分页与排序规则**
   - 区域分页固定`15条/页`，区域组最大`50条/页`
   - 排序字段严格校验白名单`ALLOW_SORT_AREA/ALLOW_SORT_GROUP`，禁止非法排序
3. **请求方式规范**
   - 页面接口：GET请求，支持缓存
   - 增删改接口：**仅支持POST请求**，JSON/表单数据兼容处理
   - 所有API接口统一返回`code+msg+data`标准JSON格式

### 4. 缓存使用规范
1. **固定缓存规则**
   - 缓存常量：`CACHE_AREA_PAGE`/`CACHE_API_LIST`等，禁止硬编码缓存时间
   - 缓存前缀：严格使用`CACHE_PREFIX`定义的键名，禁止自定义前缀
2. **缓存清理强制要求**
   - 区域增删改 → **必须调用`clear_area_cache()`**清理缓存
   - 区域组增删改 → **必须调用`clear_group_cache()`**清理缓存
   - 单条数据变更清理详情缓存，批量变更清理列表缓存
3. **页面缓存**
   - 页面视图使用`@cache_page`装饰器，API接口使用手动缓存
   - 缓存键绑定用户ID+请求参数，保证数据隔离

### 5. 权限与安全规范
1. **RBAC权限控制**
   - 权限编码固定：`area_view`(查看)、`area_add`(新增)、`area_edit`(编辑)、`area_delete`(删除)
   - 所有接口必须添加`@login_required`+`@permission_required`双装饰器
   - 区域组管理复用区域权限，无独立权限编码
2. **安全规则**
   - API接口统一添加`@csrf_exempt`豁免，适配前后端分离
   - 入参校验：名称非空、重复校验、ID合法性校验
   - 异常捕获：所有接口添加try-except，记录日志并返回友好提示

### 6. 业务逻辑核心规范
1. **区域管理规则**
   - 区域名称全局唯一，不允许重复创建
   - 已关联客户/订单的区域可删除，系统自动清空关联关系
   - 区域详情展示：客户数量、订单数量、关联区域组
2. **区域组管理规则**
   - 区域组名称唯一，支持绑定多个区域
   - 统计数据：包含区域数量、关联客户总数量
   - 编辑区域组时自动校验绑定区域的合法性
3. **统计规则**
   - 区域统计：关联客户数量
   - 区域组统计：包含区域数+客户总数，缓存10分钟

### 7. 日志规范
- 所有操作（新增/编辑/删除 区域/区域组）**必须调用`create_operation_log`**
- 操作类型固定：`create`/`update`/`delete`
- 日志记录：操作人、对象ID、名称、操作详情、IP地址

### 8. 测试与部署规范
- 测试重点：区域/组CRUD、重复名称校验、缓存清理、权限拦截、统计数据准确性
- 部署前：清理区域/组相关全部缓存
- 生产环境：禁止直接SQL修改区域/组数据，所有操作通过系统接口完成
- 关联数据变更后：强制刷新缓存，保证前端展示一致

1. **固定缓存常量（禁止修改）**
   | 缓存常量                | 作用                     | 时长  |
   |-------------------------|--------------------------|-------|
   | CACHE_AREA              | 区域数据缓存             | 1小时 |
   | CACHE_COMMON            | 商品列表/详情通用缓存    | 60秒 |
   | CACHE_SALES_RANK        | 销售排行缓存             | 10秒 |
   | CACHE_PAGINATION_COUNT  | 分页总数缓存（优化COUNT）| 120秒 |

2. **缓存前缀规范**
   - 列表缓存：`CACHE_PREFIX_PRODUCT_LIST`
   - 详情缓存：`CACHE_PREFIX_PRODUCT_DETAIL`
   - 销售排行：`CACHE_PREFIX_SALES_RANK`
   - 分页计数：`CACHE_PREFIX_PRODUCT_COUNT`
3. **缓存工具函数**
   - 统一使用`clear_product_all_cache()`清理**所有商品相关缓存**，禁止手写零散清理逻辑
4. **缓存规则**
   - 读操作（列表/详情/排行）：优先读缓存，缓存不存在则查询DB并写入缓存
   - 写操作（新增/编辑/删除/别名/导入/库存）：**必须调用缓存清理函数**
   - 缓存键：绑定搜索关键词+分页+筛选条件，保证数据隔离

### 5. 权限与安全规范
1. **RBAC权限编码固定**
   | 权限编码                | 对应功能                     |
   |-------------------------|------------------------------|
   | PERM_PRODUCT_VIEW       | 查看商品列表/详情            |
   | PERM_PRODUCT_ADD        | 新增商品                     |
   | PERM_PRODUCT_EDIT       | 编辑商品                     |
   | PERM_PRODUCT_DELETE     | 删除商品                     |
   | PERM_PRODUCT_ALIAS_ADD  | 新增商品别名                 |
   | PERM_PRODUCT_ALIAS_DELETE | 删除商品别名               |
   | PERM_PRODUCT_IMPORT     | 批量导入商品                 |
   | PERM_PRODUCT_STOCK_OP   | 快速出入库操作               |
   | PERM_PRODUCT_DETAIL     | 查看商品销售统计详情         |

2. **装饰器强制使用**
   - 所有接口**必须添加**`@permission_required(权限编码)`
   - 前后端分离接口：添加`@csrf_exempt`
   - 敏感操作：添加`@require_POST`限制请求方式
3. **数据安全规则**
   - 越权防护：无权限用户无法调用操作接口
   - 库存安全：出库数量校验，禁止超卖
   - 导入安全：重复商品自动过滤，记录失败原因

### 6. 业务逻辑核心规范
1. **商品基础规则**
   - 商品名称全局唯一，重复则创建失败
   - 默认库存：`77`，默认单位：`件`
2. **商品别名规则**
   - 一个商品可绑定多个别名，别名名称唯一
   - 搜索支持**商品名+别名**模糊匹配
3. **批量导入规则**
   - 自动识别Excel表头：商品名称、零售价、辅助单位
   - 重复商品/空数据自动跳过，记录导入结果
   - 批量创建商品，提升导入效率
4. **快速出入库规则**
   - 支持多商品同时入库/出库
   - 事务+行锁保证高并发下库存准确性
   - 出库数量不能大于当前库存
5. **商品详情规则**
   - 统计：总销量、7天销量、30天销量
   - 展示：最近销售记录、客户消费排行、专属价格
   - 所有统计使用**DB聚合查询**，性能最优
6. **销售排行规则**
   - 支持筛选：今日/本周/本月/本年 + 区域
   - 按销量/销售额排序，缓存10秒保证实时性

### 7. 日志规范
- 所有**写操作**必须调用`create_operation_log`记录日志
- 操作类型固定：`create`/`update`/`delete`/`import`/`stock_operation`
- 日志必填信息：操作人、商品ID/名称、修改前后数据、操作详情
- 日志不可删除/篡改，用于库存对账、操作追责
- 批量操作：记录总操作数量+关键商品信息

### 8. 测试与部署规范
- **测试重点**：
  1. 分页COUNT缓存生效、列表缓存自动更新
  2. 批量出入库事务回滚、库存准确性
  3. 商品/别名唯一性校验、导入功能容错
  4. 权限拦截、越权操作防护
  5. 销售统计、详情聚合数据准确性
- **部署前**：执行`clear_product_all_cache()`清空所有商品缓存，校验数据库索引
- **生产环境**：
  1. 禁止直接执行SQL修改`Product`/`ProductAlias`表数据
  2. 禁止手动修改库存，所有操作通过`quick_stock_operation`接口
  3. 定期归档历史订单，避免商品统计查询过慢

## 十一、summary 销售汇总模块专属开发规范
### 1. 代码结构规范
- **固定目录**：`summary/views.py`（汇总全量业务视图）、模板目录`summary/`、路由统一归属`summary`命名空间
- **视图函数分类**：严格按业务拆分，禁止混合编写
  1. 汇总页面：`summary_page`(商品汇总页)、`customer_summary_page`(客户汇总页)
  2. 汇总数据接口：`summary_by_group`(商品汇总)、`summary_customer_by_group`(客户汇总)
  3. 详情页面：`customer_amount_detail_page`(客户金额详情)、`product_summary_detail_page`(商品汇总详情)
  4. 详情数据接口：`get_customer_order_source`(客户订单来源)、`get_product_order_source`(商品订单来源)
  5. 导出接口：`export_product_summary`(商品导出)、`export_customer_summary`(客户导出)
  6. 公共接口：`group_list`(区域组列表)、`product_list_for_price`(商品选择列表)
- **常量统一置顶**：复用全局缓存常量`CACHE_HIGH_PRIORITY/CACHE_MID_PRIORITY`、权限常量、时间格式化规则，禁止分散硬编码
- **工具函数复用**：`parse_datetime`(时间解析)、`get_area_ids_by_group`(区域ID获取)、`create_summary_operation_log`(汇总日志)、`export_to_excel`(通用导出)全局复用，禁止重复实现

### 2. 模型（Models）关联与查询规范
1. **核心关联强制规则**
   - 汇总核心依赖：`OrderItem`→`Product/Order`、`Order`→`Customer/Area/AreaGroup`
   - 禁止修改汇总依赖模型的**联合索引**、外键关联、级联规则
2. **索引匹配强制要求（核心）**
   - 商品汇总/导出：**100%匹配**`OrderItem`统一索引`[product, order, quantity, amount]`，查询顺序严格遵循索引前缀
   - 客户汇总/导出：**100%匹配**`Order`合并索引`[status, is_settled, area, customer, create_time]`
   - 详情查询：客户订单来源匹配`Order`索引、商品订单来源匹配`OrderItem`索引
3. **查询优化强制要求**
   - 关联查询必须使用`select_related`(外键)、`prefetch_related`(一对多)，彻底消除N+1查询
   - 聚合统计必须使用`Sum/Coalesce/annotate`，`Coalesce`强制将NULL转为0，避免数值计算报错
   - 金额字段统一用`DecimalField`，返回前端时统一转为`float`并保留2位小数
   - 高频筛选：订单状态固定`status__in=['pending', 'printed', 'reopened']`，**禁止修改**排除作废订单规则
4. **区域组查询规范**
   - 统一使用`get_area_ids_by_group`极速获取区域ID列表，禁止手写区域组关联查询
   - 全部区域标识固定为`group_id=0`，禁止修改标识规则

### 3. 视图（Views）开发规范
1. **数据库性能优化**
   - 聚合查询：商品/客户汇总使用**数据库层聚合**，禁止Python内存统计
   - 详情查询：强制分页+预加载，客户/商品订单来源默认分页`10/20条/页`
   - 导出查询：复用汇总索引匹配逻辑，禁止全表无筛选导出
   - 时间处理：统一使用`parse_datetime`解析时间，自动兼容`T`分隔符格式
2. **请求与响应规范**
   - 页面入口：GET请求，渲染模板并传递基础参数
   - 数据查询/导出：POST/GET请求，统一返回`code+msg+data`标准JSON格式
   - 导出接口：直接返回Excel文件流，无JSON响应
   - 全局异常：所有接口包裹`try-except`，返回友好错误提示，状态码规范`400/500`
3. **参数校验规范**
   - 必填参数：`group_id/start_date/end_date`三要素**非空校验**，缺一不可
   - 时间格式：严格校验`YYYY-MM-DD HH:MM`，格式错误直接返回提示
   - 数值校验：分页参数转为整型，非法页码自动返回第一页
   - 导出参数：校验`selected_fields`必填，自定义字段兼容空值

### 4. 缓存使用规范
1. **固定缓存常量（复用全局，禁止修改）**
   | 缓存常量                | 作用                     | 时长  |
   |-------------------------|--------------------------|-------|
   | CACHE_HIGH_PRIORITY     | 复杂聚合查询（汇总接口） | 5分钟 |
   | CACHE_MID_PRIORITY     | 静态数据（区域组/商品列表） | 10分钟 |
2. **缓存装饰器强制使用**
   - 读接口/页面：`@cache_page`+对应缓存常量，**写操作/导出接口禁止缓存**
   - 缓存范围：区域组列表、商品选择列表、商品/客户汇总接口
3. **缓存清理规则**
   - 订单/商品/客户/区域组**写操作（增删改）**后，**手动清理汇总模块缓存**
   - 缓存键自动绑定请求参数（组ID+时间+用户），保证数据隔离
   - 详情接口无缓存，保证数据实时性

### 5. 权限与安全规范
1. **RBAC权限编码固定**
   | 权限编码                | 对应功能                     |
   |-------------------------|------------------------------|
   | PERM_ORDER_SUMMARY      | 商品/客户汇总查看、导出、详情 |
   | PERM_PRODUCT_VIEW       | 商品选择列表接口权限         |
2. **装饰器强制使用**
   - 所有接口**必须添加**`@login_required + @permission_required`双装饰器
   - 前后端分离/导出接口：添加`@csrf_exempt`豁免CSRF校验
3. **数据安全规范**
   - 按区域组权限隔离数据，普通用户仅查看授权区域汇总数据
   - 敏感数据：金额统一脱敏格式化，禁止返回原始高精度Decimal
   - 越权防护：禁止直接访问无权限的客户/商品详情

### 6. 业务逻辑核心规范
1. **商品汇总规则**
   - 按商品分组统计：总销量、总金额，按销量倒序排序
   - 仅统计有效订单，排除作废订单
   - 支持按区域组+时间范围精准筛选
2. **客户汇总规则**
   - 按客户分组统计：未结清订单总金额，按金额倒序排序
   - 固定筛选`is_settled=False`未结清订单，禁止修改
   - 展示客户名称、备注、总欠款金额
3. **详情查询规则**
   - 客户金额详情：展示该客户所有有效订单+订单商品明细
   - 商品汇总详情：展示该商品所有关联订单+购买客户/区域
   - 详情按创建时间倒序排序，展示完整订单状态
4. **Excel导出规则**
   - 支持自定义字段选择、自定义列插入
   - 自动添加总计行（总金额），样式统一（蓝色背景+白色加粗字体）
   - 文件名自动生成：`日期+汇总类型+区域组名称`
   - 列宽固定15，居中对齐，格式标准化

### 7. 日志规范
- 所有**核心操作（查询/导出/详情查看）**必须调用`create_summary_operation_log`记录日志
- 操作类型固定：`query`(查询)、`export`(导出)
- 日志必填字段：操作人、操作对象（商品汇总/客户汇总）、区域组名称、时间范围、数据条数
- 日志关联IP地址，不可删除/篡改，用于操作审计
- 导出操作单独记录，作为数据导出留痕依据

### 8. 测试与部署规范
- **测试重点**：
  1. 索引匹配：汇总查询命中联合索引，无全表扫描
  2. 数据准确性：汇总金额/数量与订单明细一致，总计计算无误
  3. 导出功能：自定义字段、样式、总计行正常生成
  4. 权限拦截：无权限用户无法访问汇总/导出/详情
  5. 缓存生效：汇总接口缓存命中，数据更新后缓存刷新
  6. 分页功能：详情接口分页正常，无数据溢出
- **部署前**：清空汇总模块缓存，校验`Order/OrderItem`联合索引完整性
- **生产环境**：
  1. 禁止直接SQL修改订单/汇总数据
  2. 禁止修改汇总索引匹配逻辑，避免性能崩溃
  3. 大时间范围汇总优先使用缓存，禁止高频全表聚合查询
